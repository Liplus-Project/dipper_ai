# Requirements

dipper_ai の要求仕様書。
本文書は「何をすべきか」を定義する。「どう実現するか」は Architecture.md を参照。

---

## 1. 目的

グローバル IP アドレスの変化を検知し、複数の DDNS サービスへ自動的に通知する常駐型クライアント。
人間がネットワーク管理を意識せずとも、DDNS の登録が常に最新に保たれる状態を維持する。

---

## 2. 基本要求

### 2.1 常駐動作

- systemd に管理される常駐プロセスとして動作すること
- 起動・停止は systemd のサービス管理コマンドで行えること
- プロセスが異常終了した場合、systemd が自動的に再起動すること
- 多重起動を防止すること

### 2.2 IP アドレス取得

- グローバル IPv4 アドレスを外部 API 経由で取得できること
- グローバル IPv6 アドレスを外部 API 経由で取得できること
- IPv4 / IPv6 の有効・無効をそれぞれ独立して設定できること
- IP 取得結果を一定期間キャッシュし、外部 API への過剰なアクセスを抑制できること（`IP_CACHE_TIME`）

### 2.3 DDNS 更新（IP 変化検知）

- 取得した IP アドレスと前回送信済み IP アドレスを比較すること
- IP アドレスが変化したエントリのみ DDNS 更新を送信すること
- **更新処理はエントリ単位で独立して行うこと**（一部エントリの失敗が他に影響しない）
- IPv4 DDNS 更新 / IPv6 DDNS 更新をそれぞれ独立して設定できること（`IPV4_DDNS` / `IPV6_DDNS`）
- IP が変化していないエントリへの不要な送信を行わないこと

### 2.4 DDNS keepalive（定期強制更新）

- **DDNS サービスは定期的な更新がない場合に登録を失効させる**
  - 失効期間はサービスによって異なる（例: MyDNS は概ね 30 日以内の更新を要求）
  - ユーザーが失効期間に合わせた間隔を自由に設定できること（`UPDATE_TIME`）
- IP が変化していなくても指定間隔で強制的に DDNS 更新を送信すること（keepalive）
- keepalive の間隔（`UPDATE_TIME`）は IP チェック間隔（`DDNS_TIME`）と完全に独立して動作すること
  - `DDNS_TIME=5m`（5 分チェック）かつ `UPDATE_TIME=30d`（30 日 keepalive）が正しく動作すること
  - `DDNS_TIME=1d`（1 日チェック）かつ `UPDATE_TIME=2m`（2 分 keepalive）が正しく動作すること
- keepalive を無効化できること（`UPDATE_TIME=0`、Cloudflare のみ利用の場合など）

### 2.5 DNS 整合性チェック

- 現在の IP アドレスと DNS 解決結果を比較できること
- 不一致を検知した場合、対象エントリの更新キャッシュをリセットして次サイクルで再送信させること

### 2.6 エラー通知

- DDNS 更新失敗をログに蓄積すること
- 蓄積されたエラーをメール通知できること（`EMAIL_ADR` が設定されている場合）
- IP 変化時の更新通知をメールで受け取れること（`EMAIL_CHK_DDNS`）
- keepalive 送信時の通知をメールで受け取れること（`EMAIL_UP_DDNS`）
- エラーチェック・通知の最小間隔を設定できること（`ERR_CHK_TIME`）

## 3. 対応 DDNS サービス

### 3.1 MyDNS.JP

- HTTP GET + Basic 認証によるアドレス通知
- IPv4 / IPv6 を個別に更新できること
- **IP 変化検知による更新・keepalive の両方に対応すること**
- 複数アカウント（エントリ）を設定できること

### 3.2 Cloudflare

- Cloudflare API トークンによる DNS レコード更新（A / AAAA レコード）
- ゾーン名からゾーン ID を自動解決すること
- **Cloudflare は API 登録が失効しないため keepalive は不要**
  - `UPDATE_TIME` の設定に関わらず Cloudflare エントリは keepalive の対象外とすること
- 複数エントリを設定できること

---

## 4. 設定要求

### 4.1 設定ファイル

- shell-style の `key=value` 形式で記述できること
- インラインコメント（`# memo`）を記述できること
- 値のダブルクォートを自動除去すること
- デフォルトのパスは `/etc/dipper_ai/user.conf` とし、環境変数 `DIPPER_AI_CONFIG` で変更できること

### 4.2 時間指定フォーマット

- 秒（`30s`）、分（`5m`）、時間（`2h`）、日（`1d`）のサフィックス形式を受け付けること
- 整数のみ（後方互換として分と解釈）も受け付けること
- 秒は分単位に切り上げること
- 最小値を下回る指定は最小値に切り上げること（エラーにしないこと）
- 負の値は最小値に切り上げること（エラーにしないこと）

### 4.3 各設定項目の要求

| 設定キー | 最小値 | 無効化 | 要求 |
|----------|--------|--------|------|
| `DDNS_TIME` | 1m | 不可（0 はデフォルト 5m） | デーモンの IP チェック間隔 |
| `UPDATE_TIME` | 3m | `0` で無効 | keepalive 送信間隔 |
| `IP_CACHE_TIME` | 15m | `0` で無効 | IP キャッシュ有効期間 |
| `ERR_CHK_TIME` | 1m | `0` で無効（毎サイクル確認） | エラーメール間隔 |

## 5. 操作インターフェース要求

### 5.1 コマンド体系

単一バイナリ `dipper_ai` にサブコマンドを持つこと。

| コマンド | 用途 | 実行タイミング |
|----------|------|---------------|
| `daemon` | 常駐動作（通常運用） | systemd が管理 |
| `update` | IP 変化時の DDNS 更新（ワンショット） | 手動 / デバッグ |
| `check` | DNS 整合性チェック（ワンショット） | 手動 / デバッグ |
| `keepalive` | 全 DDNS エントリへの強制更新（ワンショット） | 手動 / デバッグ |
| `err_mail` | エラーログ確認・通知（ワンショット） | 手動 / デバッグ |

### 5.2 daemon コマンド要求

- 起動後 10 秒待機してからチェックを開始すること（ネットワーク確立の猶予）
- 起動直後に 1 回チェックサイクルを実行すること
- `DDNS_TIME` 間隔でチェックサイクルを繰り返すこと
- `UPDATE_TIME` 間隔で keepalive を送信すること（`UPDATE_TIME=0` のときは送信しない）
- SIGTERM / SIGINT を受信したらクリーンにシャットダウンすること

---

## 6. 状態管理要求

- 状態ファイルは `STATE_DIR`（デフォルト `/etc/dipper_ai/state`）に保存すること
- ディレクトリが存在しない場合は自動生成すること
- **IP キャッシュはエントリ単位・IP ファミリ単位で個別に管理すること**
  - あるエントリの IP 変化が他のエントリの更新に影響しないこと

---

## 7. インストール・アンインストール要求

- `install.sh` による自動インストールが可能なこと
- インストール時に systemd サービスを登録・起動すること
- `uninstall.sh` による完全なアンインストールが可能なこと
- 旧バージョン（systemd タイマー方式）からのアップグレード時に旧タイマーユニットを自動削除すること

---

## 8. テスト要求

- 外部サービス（DDNS API・IP 取得 API・DNS サーバー）に依存せずにテストが実行できること
- CI（GitHub Actions）で AlmaLinux 9 / AlmaLinux 10 の両環境でテストが通ること
- ユニットテスト・統合テスト・受け入れテストの 3 層でカバーすること

---

## 9. 非要求（対象外）

- GUI による操作
- Windows / macOS のサービス管理への対応
- Cloudflare の keepalive（API 登録は失効しないため不要）
- サービスごとの keepalive 間隔の個別設定（全サービス共通 `UPDATE_TIME` で十分）
